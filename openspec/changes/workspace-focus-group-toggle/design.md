## Context

需要在文件浏览器中提供更高效的“关注组/工作区”切换入口。当前只能通过手动逐个隐藏/显示实现，缺少显式入口与一键恢复。目标是最小化对现有 UI 和数据结构的侵入，通过内部的批量隐藏能力实现。

## Goals / Non-Goals

**Goals:**
- 增加 3 个预设工作区入口，支持激活/取消激活。
- 通过批量隐藏/显示实现聚焦与快速恢复。
- 支持每个工作区配置 emoji 与 hover 文案，并持久化。
- 保持性能可接受，避免逐条文件频繁渲染抖动。

**Non-Goals:**
- 提供任意数量工作区或复杂的规则编辑器。
- 引入新的文件索引或重构现有树结构。
- 改变文件浏览器的核心排序/渲染逻辑。

## Decisions

- **入口位置与形态**：在文件浏览器顶部/标题栏附近增加紧凑按钮组，视觉与现有隐藏/显示按钮保持一致，按钮组右对齐。使用 emoji 作为主识别，hover 文案作为补充提示，占用空间小且不破坏当前布局。
- **激活行为**：点击预设工作区时，记录当前可见状态并执行批量隐藏；再次点击同一工作区时恢复到激活前的可见状态。避免用户丢失原有手动隐藏结果。
- **存储模型**：以插件设置存储 3 个工作区配置（名称、emoji、hover 文案、绑定的文件组/规则）与当前激活状态。保证重启后仍可恢复。
- **隐藏实现**：复用现有的批量隐藏机制/能力（如已有隐藏 API 或内部过滤逻辑），以单次批量操作减少渲染/重排次数。
- **状态一致性**：若用户在激活状态下手动调整隐藏，取消激活时优先恢复到激活前快照，避免混合状态造成不可预期结果。

## Risks / Trade-offs

- [隐藏快照过时] → 激活期间文件结构变动可能导致恢复不完整；记录以路径为 key，并在恢复时跳过不存在项。
- [UI 空间拥挤] → 入口区域可配置是否显示或允许折叠；必要时使用更紧凑布局。
- [性能压力] → 批量隐藏前合并操作，减少逐项更新；必要时节流或分帧更新。
